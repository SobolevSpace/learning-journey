# 设计模式

## 单例模式 (Singleton)

### 定义
确保一个类只有一个实例，并提供一个全局访问点。

### 实现方式（C++ 模板）

```cpp
template <typename T>
class Singleton {
public:
    static T& GetInstance() {
        static T instance;
        return instance;
    }

    Singleton() = default;

private:
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;
};
```

### 关键点

1. **懒初始化**：实例在第一次调用 `GetInstance()` 时才创建
2. **线程安全**：C++11 及以上标准保证局部静态变量的初始化是线程安全的
3. **禁止复制**：通过删除拷贝构造函数和拷贝赋值运算符
4. **模板泛化**：可对任何类型提供单例支持

### 使用示例

```cpp
class TestType {
public:
    float x;
    int y;

    TestType(float _x = 0.0f, int _y = 0) : x(_x), y(_y) {}
};

int main() {
    auto& instance1 = Singleton<TestType>::GetInstance();
    auto& instance2 = Singleton<TestType>::GetInstance();
    // instance1 和 instance2 是同一个对象
}
```

## 工厂模式 (Factory)

### 核心问题
对象创建的复杂性

### 主要解决的问题

1. **解耦对象创建和使用** - 客户端不需要知道具体类，只需通过工厂接口获取对象
   - 避免代码中出现大量 `new ConcreteClass()`

2. **封装创建逻辑** - 将复杂的初始化过程集中管理
   - 比如创建对象后还需要配置、依赖注入等步骤

3. **灵活扩展** - 新增产品类时，只需修改工厂或增加新工厂，无需修改客户端代码

4. **管理对象生命周期** - 工厂可以控制对象的创建、缓存、销毁

### 三种工厂模式对比

| 模式 | 工厂数量 | 适用场景 |
|------|----------|----------|
| 简单工厂 | 1个 | 产品类型少且固定 |
| 工厂方法 | 每个产品1个工厂 | 产品类型需要频繁扩展 |
| 抽象工厂 | 每个产品族1个工厂 | 多个相关产品需要一起创建 |

### 简单总结
把"怎么造"和"怎么用"分开

## 原型模式 (Prototype)

### 核心问题
通过克隆创建对象，而不是使用 `new`

### 主要解决的问题

1. **性能问题** - 创建复杂对象代价高，克隆比从头创建更快
   - 比如需要读取配置、数据库查询、大量计算等初始化步骤

2. **避免依赖具体类** - 客户端只知道原型对象，不需要知道具体类型
   - 通过 `Clone()` 获取对象，而不是 `new ConcreteType()`

3. **动态添加/删除产品类** - 运行时通过注册原型来扩展
   - 比如游戏里动态配置敌人类型，只需克隆原型即可

4. **保持初始状态** - 保存一份"标准"对象，克隆后修改

### 简单总结
复制粘贴比重新写快

## 建造者模式 (Builder)

### 核心问题
复杂对象的构建过程与表示分离

### 主要解决的问题

1. **构建步骤固定，细节可变** - 很多对象需要按固定流程创建，但每个步骤的具体实现可能不同
   - 比如组装电脑：必须按顺序安装CPU、内存、硬盘，但品牌型号可选

2. **避免构造参数爆炸** - 不需要传递十几二十个参数的构造函数
   - 用 `BuildPartA()`, `BuildPartB()` 等方法逐步构建，清晰易读

3. **支持不同表示** - 同样的构建过程可以生成不同的产品
   - 比如文本编辑器：同样的菜单操作可以生成 HTML、PDF、Markdown 等不同格式

4. **分步构建灵活性** - 可以只在需要时调用某些构建方法
   - 不像构造函数一次完成所有初始化

### 简单总结
把复杂的盖房子过程拆成一步步，同一套步骤既能盖别墅也能盖平房

## 适配器模式 (Adapter)

### 核心问题
让不兼容的接口可以协同工作

### 主要解决的问题

1. **接口不兼容** - 现有代码的接口与第三方库/旧代码接口不一致
   - 比如你的代码调用 `run()`，但库提供的是 `start()`

2. **复用现有类** - 不想修改已有代码，但需要它的功能
   - 通过适配器包装，让旧代码融入新系统

3. **统一接口** - 多个不同来源的类提供相同功能但接口不同
   - 用适配器统一成一致的调用方式

### 现实类比

- **电源适配器** - 220V 交流电 → 5V 直流电，接口不同但都能供电
- **转接头** - Type-C 转 HDMI，让不同接口的设备连接

### 简单总结
相当于中间加个"翻译官"，让两个说不同语言的人能沟通

## 桥接模式 (Bridge)

### 核心问题
将抽象部分与实现部分分离，使它们可以独立变化

### 主要解决的问题

1. **多重继承的替代方案** - 避免"类爆炸"
   - 比如有 M 种抽象类型和 N 种实现方式，传统方式需要 M×N 个子类
   - 桥接模式只需要 M+N 个类

2. **抽象与实现解耦** - 两者可以独立扩展
   - 修改实现细节不影响抽象层，反之亦然
   - 比如不同操作系统（实现）和不同UI主题（抽象）可以自由组合

3. **运行时切换实现** - 可以在运行时改变对象的实现方式
   - 不像继承要求编译时确定

4. **共享实现** - 多个抽象可以共享同一个实现对象
   - 减少对象数量，节省内存

### 现实类比

- **电视机 + 遥控器** - 遥控器是抽象，电视厂牌是实现，不同遥控器可以控制不同电视
- **形状 + 颜色** - 形状和颜色是两个独立维度，可以自由组合

### 简单总结
把两个维度的变化拆开，让它们各玩各的，不用每种组合都写一个类

## 组合模式 (Composite)

### 核心问题
统一处理单个对象和组合对象

### 主要解决的问题

1. **递归结构的统一操作** - 树形结构中，对单个节点和整个子树可以用同样的方式操作
   - 比如文件系统：文件和文件夹都可以用 `getSize()` 获取大小

2. **客户端代码简化** - 不需要判断当前处理的是单个对象还是组合
   - 同样的代码 `print(node)` 既可以打印单个文件，也可以递归打印整个文件夹

3. **层次结构的灵活管理** - 容易添加新类型的节点或重新组织层次

### 与单纯树的不同

单纯的树只是**数据结构**，组合模式是**设计模式**

| 单纯树 | 组合模式 |
|--------|----------|
| 只管理数据的层次关系 | 统一的操作接口 |
| 客户端需要区分"叶子"和"树枝" | 客户端无需区分，统一对待 |
| 叶子和树枝是不同的类 | 叶子和树枝继承同一个接口 |
| 重点是"存储"数据 | 重点是"操作"数据 |

### 简单总结
单纯树是说"有这种关系"，组合模式是说"不管什么关系，都能用同样的方式处理"

## 装饰器模式 (Decorator)

### 核心问题
在不修改原有代码的情况下，动态地为对象添加额外功能

### 主要解决的问题

1. **避免类爆炸** - 不需要为每种功能组合创建新类
   - 比如有组件和3种装饰功能，组合出 2³ = 8 种可能，如果用继承需要8个子类
   - 装饰器模式只需要 1 + 3 = 4 个类

2. **灵活组合功能** - 可以在运行时动态添加、移除功能
   - 比如咖啡：先加奶，再加糖，再加巧克力，顺序和数量都可以调整

3. **保持接口一致** - 装饰前后，对象的接口不变
   - 客户端不需要知道对象是否被装饰

4. **替代继承扩展** - 比静态继承更灵活，不破坏现有代码

### 现实类比
- **穿衣服** - 内衣 → 衬衫 → 外套 → 袍子，一层层包裹
- **汉堡包** - 面包 + 肉饼 + 芝士 + 生菜，每个都是独立的装饰层

### 简单总结
给对象"穿衣服"，想加什么功能就套一层，不修改原对象

## 外观模式 (Facade)

### 核心问题
为复杂的子系统提供一个简单的统一入口

### 主要解决的问题

1. **简化客户端调用** - 不需要直接调用多个子系统的复杂接口
   - 比如启动电脑：客户端不需要知道 BIOS、引导程序、系统加载等细节
   - 只需调用 `boot.start()` 一个方法

2. **解耦客户端与子系统** - 客户端不依赖子系统的具体实现
   - 子系统内部变化不影响客户端代码

3. **分层架构** - 隐藏底层复杂性，提供清晰的接口层次
   - 客户端只关心高层接口，不需要了解底层细节

4. **减少客户端的依赖** - 客户端只需依赖外观类，而不是多个子系统
   - 降低耦合度，提高代码的可维护性

### 现实类比
- **餐厅点餐** - 你只需告诉服务员"套餐A"，不需要自己去厨房下单菜、端饮料
- **汽车启动** - 按一个键就能启动，不需要知道点火、喷油、启动电机等过程

### 简单总结
给复杂系统加个"前台"，客户端只和前台打交道，不用管后面的复杂流程

## 代理模式 (Proxy)

### 核心问题
为对象提供一个代理以控制对该对象的访问

### 主要解决的问题

1. **访问控制** - 限制或监控对真实对象的访问
   - 权限检查：只有通过验证才能访问
   - 访问日志：记录每次访问的时间、操作者等信息

2. **远程代理** - 访问不在同一地址空间的对象
   - 比如 RPC 调用：客户端代理负责网络通信，客户端感觉就像本地调用

3. **延迟加载** - 对象创建开销大，用代理延迟创建
   - 图片懒加载：先显示占位符，真正显示时才加载大图

4. **缓存代理** - 缓存昂贵操作的结果
   - 数据库查询代理：缓存查询结果，减少数据库压力

5. **智能引用** - 增加对对象的额外操作
   - 引用计数、垃圾回收、写时复制等

### 现实类比
- **明星经纪人** - 想请明星演出，先找经纪人，处理预约、费用、档期等
- **网络代理** - 访问受限资源通过代理服务器中转

### 简单总结
给真实对象找个"替身"，在中间把关、打杂，真实对象只干活

## 享元模式 (Flyweight)

### 核心问题
通过共享来支持大量细粒度对象的重用，减少内存占用

### 主要解决的问题

1. **内存节省** - 大量相似对象时，只保存一份共享状态
   - 比如文本编辑器中字符"A"出现1000次，只用1个字符对象

2. **内外分离** - 将对象状态分为内部状态（可共享）和外部状态（不可共享）
   - 内部状态：如字符形状、颜色等可共享的数据
   - 外部状态：如位置、大小等每个实例不同的数据

3. **性能优化** - 减少对象创建和垃圾回收的开销
   - 对象池管理，重复利用已创建的对象

### 现实类比
- **字体缓存** - 系统只加载一份字体，所有文本共享
- **游戏中的树** - 1000棵树共享同一个树模型，只有位置不同

### 简单总结
能共享的就共享一份，不能共享的每个对象各自带，省内存

## 责任链模式 (Chain of Responsibility)

### 核心问题
将请求的发送者和接收者解耦，让多个对象都有机会处理请求

### 主要解决的问题

1. **免于耦合** - 发送者不需要知道具体谁会处理请求
   - 请求沿着链传递，直到有对象处理为止

2. **灵活的处理链** - 可以动态调整处理顺序或增删处理器
   - 比如日志级别：DEBUG -> INFO -> WARN -> ERROR，可以灵活配置

3. **责任分散** - 每个处理器只处理自己能处理的部分
   - 不用一个对象处理所有情况，代码更清晰

4. **请求流程可变** - 不同的请求可能有不同的处理路径
   - 同一条链，不同请求可能在不同节点被处理

### 现实类比
- **公司审批流程** - 主管 -> 经理 -> 总监 -> CEO，逐级审批，谁能批就停在谁那里
- **异常处理** - try-catch 块的嵌套，异常一层层向上抛

### 简单总结
像传纸条一样，没人处理就往下传，有人处理就停下
